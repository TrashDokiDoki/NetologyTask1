# Задача 2
Выберите один из вариантов платформы в зависимости от задачи. Здесь нет однозначно верного ответа так как все зависит от конкретных условий: финансирование, компетенции специалистов, удобство использования, надежность, требования ИБ и законодательства, фазы луны.

**Тип платформы:**

1. физические сервера;
2. паравиртуализация;
3. виртуализация уровня ОС;
 
 **Задачи:**
 
1. высоконагруженная база данных MySql, критичная к отказу;
2. различные web-приложения;
3. Windows-системы для использования бухгалтерским отделом;
4. системы, выполняющие высокопроизводительные расчёты на GPU.
5. Объясните критерии выбора платформы в каждом случае.

**Решение:**

 1. Высоконагруженная БД MySQL, критичная к отказу
 Выбор: паравиртуализация (например, Xen с PV-драйверами или KVM с virtio).
 Обоснование:

 БД требует высокой производительности ввода-вывода и минимальных задержек. Паравиртуализация даёт производительность, близкую к физической, за счёт оптимизированных драйверов и меньших накладных расходов по сравнению с полной эмуляцией.

 При этом сохраняется возможность live-миграции и автоматического перезапуска при сбоях (в отличие от физических серверов, где отказ оборудования часто приводит к длительному простою).

 Можно использовать кластерные решения (Galera, репликация) поверх виртуальных машин, а гипервизор обеспечит изоляцию ресурсов (CPU, память, IOPS).

 Виртуализация уровня ОС (контейнеры) не подходит из-за общего ядра и ограничений по управлению памятью/дисковым кешем, что критично для БД.

2. Различные web-приложения
Выбор: виртуализация уровня ОС (контейнеры, например, Docker/LXC).
Обоснование:

 Web-приложения обычно хорошо масштабируются горизонтально, не требуют высокой изоляции, часто работают на Linux.

 Контейнеры обеспечивают быстрый запуск, минимальные накладные расходы по CPU/памяти, удобное управление через оркестраторы (Kubernetes, Swarm).

 Упрощается CI/CD, обновление приложений, утилизация ресурсов – на одном хосте можно запустить десятки контейнеров.

 Если в составе есть Windows-компоненты, контейнеризация Windows также возможна (Windows Containers), но в общем случае предпочтение отдаётся Linux-контейнерам.

3. Windows-системы для бухгалтерского отдела
Выбор: паравиртуализация (или полная виртуализация с паравиртуальными драйверами) – например, Hyper-V, VMware ESXi или KVM с virtio-драйверами для Windows.
Обоснование:

 Windows не поддерживает контейнеризацию уровня ОС в привычном смысле (требуется полноценное ядро), поэтому нужна полноценная виртуализация.

 Паравиртуализация даёт хорошую производительность для Windows (особенно при использовании интегрированных компонентов Hyper-V или драйверов virtio).

 Обеспечивается надёжная изоляция – бухгалтерские системы критичны к безопасности и стабильности.

 Физические сервера были бы избыточны, так как задачи не являются высоконагруженными, а управление виртуальными машинами упрощает резервное копирование и миграцию.

4. Системы с высокопроизводительными расчётами на GPU
Выбор: физические сервера (bare-metal) с прямым доступом к GPU.
Обоснование:

 GPU-расчёты критичны к задержкам и пропускной способности шины PCIe. Любой уровень виртуализации добавляет накладные расходы (даже при PCIe-passthrough).

 Хотя современные гипервизоры (KVM, VMware) поддерживают проброс GPU с производительностью, близкой к нативной, это требует сложной настройки и не всегда предсказуемо.

 Физический сервер даёт максимальную производительность и упрощает управление драйверами NVIDIA/CUDA.

 Если требуется гибкость (например, несколько сред для разных проектов), можно использовать физические серверы с возможностью загрузки разных ОС или использовать контейнеризацию на уровне ОС (например, NVIDIA Docker), но для критичных расчётов всё же предпочтительнее выделенный физический узел.

# Задача 3
Выберите подходящую систему управления виртуализацией для предложенного сценария. Опишите ваш выбор.

Сценарии:

1. 100 виртуальных машин на базе Linux и Windows, общие задачи, нет особых требований. Преимущественно Windows based-инфраструктура, требуется реализация программных балансировщиков нагрузки, репликации данных и автоматизированного механизма создания резервных копий.
2. Требуется наиболее производительное бесплатное open source-решение для виртуализации небольшой (20-30 серверов) инфраструктуры на базе Linux и Windows виртуальных машин.
3. Необходимо бесплатное, максимально совместимое и производительное решение для виртуализации Windows-инфраструктуры.
4. Необходимо рабочее окружение для тестирования программного продукта на нескольких дистрибутивах Linux.

 **Решение:**


Сценарий 1: 100 ВМ (Linux + Windows), преимущественно Windows, нужны балансировщики, репликация, автоматический бэкап
Выбор: Microsoft Hyper-V (в составе Windows Server + System Center Virtual Machine Manager).
Обоснование:

  Нативная поддержка Windows-гостей, максимальная совместимость и производительность для Windows.

  Встроенные средства репликации (Hyper-V Replica), балансировки (NLB, или интеграция с Azure Load Balancer), планировщик резервного копирования (Windows Server Backup, Azure Backup).

  SCVMM даёт централизованное управление большим количеством ВМ, автоматизацию, мониторинг.

  Альтернатива – VMware vSphere, но он дороже и менее интегрирован с Windows-средой.


Сценарий 2: бесплатное open-source решение для 20–30 серверов (Linux/Windows), производительность важна
Выбор: Proxmox VE (на базе KVM и LXC).
Обоснование:

  Полностью бесплатный, с открытым исходным кодом, имеет удобный веб-интерфейс.

  KVM обеспечивает высокую производительность как для Linux, так и для Windows (при установке драйверов virtio).

  Встроенные функции: live-миграция, кластеризация, резервное копирование, репликация на уровне хранилища (ZFS, Ceph).

  Proxmox активно развивается, имеет большое сообщество, подходит для инфраструктуры среднего размера.


Сценарий 3: бесплатное, максимально совместимое и производительное решение для Windows-инфраструктуры
Выбор: Hyper-V Server (бесплатная версия гипервизора Microsoft).
Обоснование:

  Это специализированная, бесплатная ОС, содержащая только роль гипервизора, без графического интерфейса.

  Обеспечивает наилучшую совместимость с Windows-гостями, поддержку динамической памяти, live-миграцию, репликацию.

  Управление осуществляется удалённо через Windows Admin Center или PowerShell – это бесплатно.

  В отличие от бесплатного ESXi (ограничение по vCPU), Hyper-V Server не имеет таких ограничений на количество ядер.


Сценарий 4: окружение для тестирования ПО на нескольких дистрибутивах Linux
Выбор: VirtualBox (бесплатный, кроссплатформенный).
Обоснование:

  Простота установки и использования на рабочих станциях.

  Поддерживает практически все дистрибутивы Linux, позволяет создавать снимки (snapshots) для быстрого возврата к состояниям.

  Хотя производительность ниже, чем у KVM, для тестирования этого достаточно.

  Альтернативно можно использовать Vagrant с провайдером VirtualBox для автоматизации развёртывания нескольких ВМ.

# Задача 4
Опишите возможные проблемы и недостатки гетерогенной среды виртуализации (использования нескольких систем управления виртуализацией одновременно) и что необходимо сделать для минимизации этих рисков и проблем. Если бы у вас был выбор, создавали бы вы гетерогенную среду или нет?

**Решение:**

**Основные недостатки**
 
 1. Сложность управления – разные гипервизоры имеют свои инструменты (vCenter, SCVMM, Proxmox, CLI), что увеличивает время администрирования и требует разнородных навыков.

 2. Отсутствие единой политики безопасности – настройки доступа, шифрования, сетевых правил могут различаться, что создаёт «слепые зоны».
 
 3. Проблемы миграции – live-миграция между разными платформами невозможна; требуется конвертация форматов (VMDK, VHD, QCOW2) и остановка ВМ.
 
 4. Сложность мониторинга – каждый гипервизор предоставляет свои метрики, интеграция в единую систему мониторинга затруднена.
 
 5. Конфликты ресурсов – разные планировщики могут по-разному распределять ресурсы, особенно при совместном использовании общего хранилища или сети.
 
 6. Увеличение затрат – требуется больше лицензий (если используются коммерческие продукты), больше времени на обучение и поддержку.
 
**Минимизация рисков**
 1. Стандартизация форматов образов – использовать общий формат (например, OVF) и конвертеры (qemu-img, StarWind V2V) для миграции.
 
 2. Централизованное управление – внедрить платформу оркестрации (OpenStack, CloudStack, или VMware vRealize), которая может абстрагироваться от конкретного гипервизора через драйверы.
 
 3. Единое хранилище – использовать общую СХД или распределённое хранилище (Ceph, GlusterFS), доступное всем гипервизорам.
 
 4. Автоматизация – применять Infrastructure as Code (Terraform, Ansible) для единообразного развёртывания ВМ на любой платформе.
 
 5. Унифицированный мониторинг – использовать Prometheus + экспортёры для сбора метрик со всех гипервизоров, а также логирование в единую систему (ELK).
 
 6. Обучение персонала – создать документацию по каждой платформе и регламенты взаимодействия.
 
 **Личное мнение**
 Если есть выбор, я бы не создавал гетерогенную среду без крайней необходимости. Она оправдана только в исключительных случаях:

1. при поэтапной миграции с одной платформы на другую;

2. при наличии устаревшего оборудования, которое не поддерживает современный гипервизор;

3. при специфических требованиях заказчика (например, обязательное использование Hyper-V для Windows-приложений и KVM для высокопроизводительных Linux-задач).

 В большинстве случаев единая платформа (например, VMware или Proxmox) обеспечивает достаточную гибкость, упрощает администрирование и снижает совокупную стоимость владения. Гетерогенность стоит применять только при явном выигрыше в производительности, стоимости или функциональности, тщательно взвесив все риски.
